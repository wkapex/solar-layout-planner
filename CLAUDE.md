# 野立て・陸屋根・傾斜屋根 太陽光 配置平面図ツール

ブラウザ完結型の太陽光パネル配置図 作成ツール。敷地図（公図PDF・設計図・航空写真）を
読み込み、縮尺校正 → 敷地作図 → パネル自動配置 → DCストリング結線 → DXF/PDF出力までを行う。
野立てはフェンスライン（境界離隔）と配置可能ライン（フェンス+2m内側）を自動生成する。
プロジェクト一式（背景画像・校正・敷地・条件・PCS）をJSONに保存し再開できる。

## 重要な前提・設計思想

- **外部APIに依存しないローカル完結アプリ**（ジオコーディングのGSI住所検索のみ例外）。
  ブラウザ内で全処理が完結し、サーバーを持たない。
- **座標系**: 内部処理はすべて「数学メートル座標（x右・y上、単位m）」で統一。
  画像ピクセル座標（x右・y下）との相互変換は `polygon.ts`（imgToMeters / metersToImg）に集約。
  この変換規約を崩さないこと。新しい描画・出力も必ずこの規約に従う。
- **DXF出力はmm単位**（×1000）。JWCAD等がmmで原寸解釈するため。R12 ASCII形式。
- **ビルド**: Vite + TypeScript。`npm run dev` で開発サーバー、`npm run build` で本番ビルド。

## 公開リポジトリに関する注意

## モジュール構成（src/）

| ファイル | 役割 |
|---|---|
| `main.ts` | UI統合・状態管理・各ステップの司令塔。DOMイベント接続 |
| `canvas.ts` | 画像のズーム/パン表示、オーバーレイ描画、クリック→画像座標変換 |
| `imageLoader.ts` | PDF(pdfjs-dist)/画像の読込・ラスタライズ |
| `calibration.ts` | 縮尺校正（実距離2点 / 公図座標2点） |
| `jpr.ts` | 平面直角座標系（JGD2011）。公図座標→緯度経度 |
| `polygon.ts` | 敷地多角形と px⇔m 座標変換（★変換規約の中心） |
| `geometry.ts` | 2Dベクトル演算・多角形内外判定・点線分距離など基礎 |
| `geocode.ts` | 住所→緯度経度（国土地理院API） |
| `solar.ts` | suncalc で冬至10-14時の太陽位置→影離隔計算の入力 |
| `panels.ts` | パネル仕様の手動入力＋localStorage履歴 |
| `layout.ts` | **パネル自動配置エンジン**。A:真南優先 / B:敷地なり最大 / ROOF:屋根。位相探索＋細密u走査で最大化。`arrayComposition`（凡例用のアレイ構成文字列）もここ |
| `summary.ts` | 配置結果の集計HTML生成 |
| `stringing.ts` | **DCストリング結線エンジン**。Ns枚=1回路の切り出し・PCS回路割当・PCS別配色・ラベル位置（`labelPos`/`labelDir`）・PCS囲い（`groupPcsBoxes`） |
| `dxfExport.ts` | **DXF出力**（mm単位）。SITE/FENCE/PANEL/NOTE/STRING_PCS{n}/PCS_BOX/POLE レイヤー |
| `pdfExport.ts` | **PDF出力**。A4横に最大スケールで中央配置し1枚画像化（外部ライブラリ非依存） |
| `project.ts` | **プロジェクト保存/再開**。背景画像(dataURL)＋全入力条件をJSON1ファイルに |

## 主要な処理フロー

```
読込(imageLoader) → 校正(calibration/jpr) → 敷地作図(polygon)
  → パネル配置(layout: フェンス=insetPolygon, 配置はフェンス+2m内側)
  → パワコン指定 → 結線生成(stringing) → DXF/PDF出力(dxfExport/pdfExport)
  →（任意）プロジェクト保存/再開(project)
```

## DCストリング結線（stringing.ts）の仕様

- 1ストリング = 直列数(Ns)枚 = 1回路。PCS容量 = 並列×MPPT入力数（回路数）。
- たどり方: **アレイ内完結が大前提**。端数で完結できない場合のみ隣アレイへ連続（蛇行）。
- 割当: アレイを帯(v)→左右(u)の読み順で連結し、空間順にPCSの回路へ詰める。
- 表記: 同一PCSは同色（`PCS_COLORS`）。回路ラベル `PCS{台}-{回路}` は1ストリング1個、
  主要段の横中央×段のすぐ外（上段=上/下段=下）。重なりは labelDir 方向のみへ回避。
- PCS設置位置はアレイごとに四角で囲い「PCS×n」表記（`groupPcsBoxes`）。1台ごとの記号は出さない。

## 図面表記（PDF/DXF共通）

- 敷地境界=オレンジ、フェンスライン=緑（野立てのみ。境界離隔ぶん内側、延長mも凡例に記載）。
- 先方柱=灰色の塗り丸＋ラベル（クリック指定・任意）。
- 凡例（PDF下部・26px）: パネル枚数/kW・アレイ構成（○段○列×○アレイ）・モジュール型番・
  設置角度（パターンBのみ、真南=0度）・南北アレイ間離隔・フェンス延長・結線サマリ。
- アレイごとの段列表記は試行の結果**廃止**（大規模で読みにくい）。凡例のアレイ構成に集約。

## 既知の制約・今後の課題

- Nsはサイト共通前提（PCSごとに異なる直列数は未対応・警告のみ）。
- 総枚数がNsの倍数でない場合は端数ストリングとして警告。
- **将来構想**: Claude API を呼び、図面を解析してAIが結線・配置を提案する拡張。
  その際は本ツールのデータ構造（数学m座標のポリライン）をそのまま活かせる設計になっている。

## 開発時の注意

- TypeScript strict モード。`npm run build`（tsc）で型チェックが通ることを確認。
- `imageLoader.ts` の `pdf.worker.mjs?url` は Vite 専用構文。tsc 単体では解決できないが
  vite build では正常に通る（既知・正常）。
- ブラウザの localStorage を使用（パネル履歴）。artifactsではなく通常のWeb環境で動作。
